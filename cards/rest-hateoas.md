# Что такое HATEOAS в REST API?

## Ответ

**HATEOAS** (Hypermedia As The Engine Of Application State) — принцип, по которому ответ API содержит не только данные, но и **ссылки на доступные дальнейшие действия**. Клиенту не нужно заранее знать все URL — он «переходит по ссылкам», как пользователь по сайту.

```json
{
  "id": 42,
  "status": "pending",
  "total": 1500,
  "_links": {
    "self":   { "href": "/orders/42" },
    "cancel": { "href": "/orders/42/cancel", "method": "DELETE" },
    "pay":    { "href": "/orders/42/pay", "method": "POST" }
  }
}
```

Если заказ уже оплачен, ссылки `cancel`/`pay` в ответе просто не появятся — клиент узнаёт о доступных переходах из самого ответа, а не хардкодит логику «что можно сделать с заказом в этом статусе».

**Без HATEOAS** (так устроено большинство реальных API):

```json
{ "id": 42, "status": "pending", "total": 1500 }
```

Клиент сам должен знать по документации, что для отмены нужно вызвать `DELETE /orders/42/cancel`, и сам проверять, разрешено ли это в текущем статусе.

**Важно:** HATEOAS — часть исходного описания REST от Roy Fielding, но на практике большинство API называют себя «RESTful», не реализуя это ограничение — это нормально и не считается ошибкой, если команда сознательно выбрала более простой подход.
