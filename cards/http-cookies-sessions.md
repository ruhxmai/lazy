# Как работают cookies и sessions в вебе?

## Ответ

**Cookie** — небольшой кусок данных, который сервер просит браузер сохранить и присылать обратно с каждым запросом на этот домен.

```
# Сервер отправляет
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600

# Браузер автоматически добавляет в каждый следующий запрос
Cookie: sessionId=abc123
```

**Атрибуты:**
- `HttpOnly` — недоступна из JS (`document.cookie`), защита от XSS
- `Secure` — передаётся только по HTTPS
- `SameSite=Lax/Strict/None` — защита от CSRF, ограничивает отправку с чужих доменов

**Session-based auth:**

```
Клиент → POST /login → Сервер создаёт запись в хранилище сессий (Redis/DB)
                        и отдаёт Set-Cookie: sessionId=abc123
Клиент → GET /profile (Cookie: sessionId=abc123) → Сервер ищет сессию по id → отдаёт данные
```

Сервер хранит **состояние** (state) — сессию можно мгновенно отозвать, удалив запись.

**Отличие от JWT:** JWT — самодостаточный токен, сервер ничего не хранит (stateless), но отозвать его до истечения `exp` сложнее (нужен blacklist).

| | Session + cookie | JWT |
|---|---|---|
| Хранение состояния | На сервере | В самом токене |
| Моментальный logout/отзыв | Просто | Сложно |
| Масштабирование (много серверов) | Нужно общее хранилище сессий | Не нужно |
