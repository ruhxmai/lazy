# Как правильно версионировать REST API?

## Ответ

Версионирование нужно, чтобы вносить **breaking changes** (менять формат ответа, убирать поля), не ломая клиентов, которые ещё используют старую версию.

**1. URI-версионирование** — самый распространённый и наглядный способ:

```
GET /api/v1/users
GET /api/v2/users
```

**2. Версия в заголовке** — URL остаётся чистым:

```http
GET /users
Accept: application/vnd.myapp.v2+json
```

**3. Версия в query-параметре** — просто, но легко забыть указать:

```
GET /users?version=2
```

| Способ | Плюсы | Минусы |
|---|---|---|
| URI (`/v1/...`) | Просто, видно в логах, легко тестировать в браузере | «Грязный» URL, ресурс формально меняет адрес |
| Header | Чистый URL | Не виден при простом переходе по ссылке, сложнее дебажить |
| Query param | Просто добавить | Легко забыть, путается с обычными параметрами фильтрации |

**Рекомендация:** для большинства проектов достаточно URI-версионирования (`/api/v1/...`). Меняйте major-версию только при breaking changes — добавление нового необязательного поля версии не требует.
