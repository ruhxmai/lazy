# Как версионировать REST API?

## Ответ

Версионирование нужно, чтобы менять API (ломающие изменения структуры ответа, полей, поведения), не ломая существующих клиентов.

**1. URI-версионирование** — самый частый и явный способ:

```
GET /api/v1/users/42
GET /api/v2/users/42
```

**2. Версия в заголовке:**

```http
GET /users/42
Accept-Version: v2
```

**3. Версия через content negotiation (`Accept`):**

```http
GET /users/42
Accept: application/vnd.myapp.v2+json
```

**4. Версия в query-параметре** (проще всего, но менее «чисто»):

```
GET /users/42?version=2
```

| Способ | Плюсы | Минусы |
|---|---|---|
| URI (`/v1/...`) | видно в логах, легко тестировать в браузере | «замусоривает» URL, URI формально должен быть постоянным идентификатором ресурса |
| Заголовок | чистый URL | неудобно тестировать вручную, менее заметно |
| `Accept` header | правильно с точки зрения REST/HATEOAS | сложнее в реализации и отладке |

**Практика:** большинство публичных API (Stripe, GitHub, Twitter) используют URI- или header-версионирование. Важно поддерживать старую версию какое-то время (**deprecation period**) и явно предупреждать клиентов через заголовок `Deprecation` или `Sunset`.
