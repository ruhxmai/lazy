# Virtual DOM и Reconciliation

**Virtual DOM** — это облегчённое представление реального DOM в виде обычных JS-объектов. Каждый раз, когда меняется состояние компонента, React строит новое виртуальное дерево, сравнивает его с предыдущим (**diffing**) и применяет к настоящему DOM только минимальный набор изменений (**patching**).

## Зачем это нужно

Прямая работа с реальным DOM дорогая: каждое изменение может вызвать **reflow** (пересчёт геометрии страницы) и **repaint** (перерисовку пикселей). Если бы React на каждое изменение состояния перестраивал весь DOM с нуля, приложение с сотнями элементов заметно тормозило бы.

Virtual DOM решает это иначе: изменения сначала происходят в памяти (дёшево), затем вычисляется **минимальный diff**, и только этот diff применяется к настоящему DOM (дорого, но по минимуму).

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div className="counter">
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}
```

При каждом клике React не пересоздаёт `<div>` и `<button>` заново — он сравнивает новое виртуальное дерево со старым и обновляет в реальном DOM только текстовый узел внутри `<p>`.

## Как работает diffing (reconciliation)

Полное сравнение двух произвольных деревьев — задача с сложностью O(n³), что для UI неприемлемо. React использует эвристический алгоритм с двумя допущениями:

1. **Разные типы элементов дают разные деревья.** Если тег сменился (`<div>` → `<span>`), React не пытается найти сходства — старое поддерево удаляется, новое строится с нуля вместе со всем состоянием внутри.
2. **Списки сравниваются по `key`.** Разработчик подсказывает React, какие элементы списка соответствуют друг другу между рендерами, через проп `key`.

```jsx
// Без key или key={index} — React сопоставляет элементы по позиции.
// При вставке элемента в начало массива React решит, что изменились
// ВСЕ элементы, и перерисует лишнее.
todos.map((todo, i) => <TodoItem key={i} {...todo} />)

// key={todo.id} — React находит тот же DOM-узел по стабильному id
// и просто переставляет его, а не пересоздаёт.
todos.map((todo) => <TodoItem key={todo.id} {...todo} />)
```

## Fiber: почему это ещё и про приоритеты

Начиная с React 16, движок реконсиляции переписан на архитектуру **Fiber**. Главное отличие от старого «стек»-алгоритма: работу по обходу дерева можно **прерывать и возобновлять**. Это позволяет React:

- разбивать рендеринг на маленькие единицы работы (fiber-узлы);
- отдавать управление браузеру между единицами, чтобы не блокировать анимации и ввод пользователя;
- назначать разный приоритет обновлениям (клик пользователя важнее, чем фоновая загрузка данных).

## Batching

React группирует несколько вызовов `setState` в одном обработчике события в **одно** обновление и один проход diffing/patching, а не запускает пересчёт на каждый вызов по отдельности.

```jsx
function handleClick() {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // React выполнит только один re-render, а не два
}
```

```mermaid

    A[Изменение state / props] --> B[Строится новое Virtual DOM дерево]
    B --> C{Diffing: сравнение\nс предыдущим деревом}
    C -->|тип узла не менялся| D[Сравнить атрибуты и детей]
    C -->|тип узла сменился| E[Удалить старое поддерево,\nпостроить новое]
    D --> F[Список изменений\n"patch"]
    E --> F
    F --> G[Применить patch\nк реальному DOM]
    G --> H[Браузер: reflow + repaint\nтолько для изменённых узлов]
```

## Компромиссы

Virtual DOM — не бесплатная магия и не всегда быстрее ручной оптимизации: сам diffing тоже требует вычислений. Для очень простых точечных обновлений (например, обновление одного текстового поля в jQuery) прямая работа с DOM может быть быстрее. Выгода Virtual DOM — не в сыром быстродействии, а в том, что он даёт **декларативную модель** («опиши, как должен выглядеть UI при данном состоянии») при сохранении приемлемой производительности без ручного управления каждым DOM-узлом.
